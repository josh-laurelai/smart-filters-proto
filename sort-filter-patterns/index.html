Sort & Filter in Authoring Interfaces

Research into how products handle the tension between data organization (sorting/filtering) and authoring stability (entries staying where you put them).

The Core Problem

When users are authoring data (adding/editing timesheet entries), they need spatial stability — "I put it here, it stays here." When users are reviewing data (scanning for errors, checking totals), they benefit from sorting and filtering. These two needs conflict when both happen in the same view.

The problem is amplified in timesheets because entries can be auto-generated (from timers, calendar events) while the user is actively working in a filtered view. An entry that vanishes after creation — or teleports mid-edit — erodes trust in the interface.

Jobs to Be Done: Why Sort/Filter a Timesheet?

Before picking a pattern, we need to ask why someone would sort or filter their timesheet. The answer shapes which solution fits.

Job Trigger What They Actually Need
Fix validation errors before releasing "I see 2 errors in the filter bar, let me find them" Surface the broken entries — not necessarily re-sort everything
Review entries by status (released vs. pending) End-of-day or end-of-week review See what's been released and what hasn't — grouping/filtering
Find a specific entry to edit "I need to change the description on that Cahill Gordon entry" Search or scan — often faster than sort/filter
Check if hours look right "Did I really spend 4 hours on that?" Sort by duration — but only momentarily
Focus on one project "Let me just look at RAD Research entries" Already handled by project grouping

The dominant job is "find and fix things that need attention" — not "permanently re-order my timesheet." This suggests the solution may not be better sorting, but better attention surfacing.

How Other Products Handle This

Google Sheets / Excel

One-Shot Sort

Sorting is a destructive one-time action — rows rearrange, then the view returns to manual order. No "live sort." Filtering is separate: hides rows without moving anything. Users can toggle filters on/off.

Airtable

Auto-Sort Toggle

"Automatically sort records" toggle. When ON: live-sort, drag-reorder disabled, sorted columns get orange highlight. When OFF: sort applied once, manual reorder available, re-sort on demand.

Linear / Todoist

Manual is Default

"Manual" is an explicit sort option. When any other sort is active, drag-to-reorder is disabled. Clear either/or: your order or the system's order.

Notion / Monday.com

Smart Defaults

New items in a filtered/grouped view inherit the group's values (e.g., creating under "Released" sets status to Released). Prevents the "vanishing new item" problem.

Three Variants to Explore

Based on the JTBD analysis and product research, we designed three approaches. Each addresses the core problem differently — click the tabs above to see them live.

Variant Core Idea Best For
A: Needs Attention Don't sort — surface problems inline. A "needs attention" bar collects broken entries at the top of each project group without disrupting order. If the primary JTBD is "fix errors before release" and full sort/filter is rarely needed
B: One-Shot Sort Sort is an action, not a mode. Click sort → entries rearrange → view returns to manual. Filter shows a banner when items are hidden. If users need real sorting occasionally but spatial stability matters more
C: Auto-Sort Toggle Airtable-style toggle for "keep sorted" with strong visual indicators. When off, sort is one-shot. When on, live-sort with drag disabled. If the vocal minority needs persistent sort and you want maximum flexibility

Variant A: Needs Attention

Instead of sorting to find problems, surface them inline. A collapsible "needs attention" bar appears at the top of each project group when entries have validation errors or missing data — keeping the default order stable.

Tradeoffs

Strengths Directly addresses the #1 JTBD (fix errors before release). No spatial disruption — entries stay where they are. Zero learning curve. The attention bar doubles as a progress indicator ("0 entries need attention" = you're done). Works within existing project grouping.
Weaknesses Doesn't address less common JTBDs (sort by hours, filter by status). The "needs attention" concept needs a clear definition — what counts? Only works well when the system can identify problems automatically. Vocal power users who want full sort/filter won't get it here.

Tradeoffs

Strengths Familiar mental model (spreadsheet-like). Spatial stability after sort — entries don't keep moving. Clear "sorted" state with easy reset. Filter banner prevents the "where did my entry go?" confusion. Supports all JTBDs (sort by hours, filter by status, etc.).
Weaknesses "One-shot sort" is subtly different from what users expect if they come from Airtable/Notion where sorts persist. The sort banner adds visual noise. Users might not realize they can manually reorder after sorting. Requires new entries to appear at top (outside sort order) to avoid confusion.

Variant C: Auto-Sort Toggle

Airtable-inspired approach: sort rules can be set, and a "Keep sorted" toggle controls whether sorting is live (entries re-sort as you edit) or manual (sort once, then freely reorder). Strong visual indicators show when auto-sort is active.

Tradeoffs

Strengths Most powerful and flexible option. Satisfies the vocal minority who wants persistent sorting. The toggle makes the tradeoff explicit — users choose between stability and auto-organization. Strong visual indicators (orange highlights, disabled drag handles, banner) prevent confusion. "X of Y shown" per project group prevents the "where did it go?" problem.
Weaknesses Highest complexity — more UI surface, more states to understand. The "keep sorted" concept may be foreign to users who haven't used Airtable. Multiple banners (auto-sort + filter) take significant vertical space. Risk of over-engineering for what may be a niche need. Disabled drag handles might feel restrictive.
